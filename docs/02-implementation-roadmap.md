# YOLOv13 雨霧天氣系統 - 實施路線圖 (Implementation Roadmap)

## 1. 里程碑定義與當前狀態

本系統採用迭代分階段實現，目前前四項基礎里程碑已陸續完工，當下進度位於切換至 YOLOv13 真實推理服務與效能升級的交界口。

| 階段 (Phase) | 核心目標 | 當前狀態 | 結果說明 |
| --- | --- | --- | --- |
| **Phase 0** | 架構設計與文檔定義 | ✅ **已完成** | 系統設計、API 契約與資料表結構定義完畢。 |
| **Phase 1** | 基礎骨架與環境搭建 | ✅ **已完成** | Django 後端, FastAPI 推理端, React 前端專案初始化成功，MySQL/Redis 基礎建設啟動。 |
| **Phase 2** | 核心業務閉環與 Mock 推理 | ✅ **已完成** | 影像上傳、任務拋轉、Mock Adapter 推論、結果明細入庫與前端畫面渲染全數打通驗證。 |
| **Phase 3** | 系統擴展與 Redis 快取優化 | ✅ **已完成** | 首頁 Dashboard 快取統計、系統配置中心設計與操作審計日誌建置完畢。 |
| **Phase 4** | **真實 YOLOv13 接入** | 🔄 **進行中 (代碼就緒)** | FastAPI 的 `yolov13.py` Adapter 已經實作就緒。**現狀與差距**：代碼本身可執行，唯依賴開發者在本地備妥 `.pt` 權重模型檔並切換啟動變數。 |
| **Phase 5** | 異步隊列與工程化部署 | 📅 **計畫中** | 引入 Celery 解耦推論等待；Docker 完整生產級容器化編排部署。 |

## 2. 後續計畫與依賴 (Next Steps)

針對目前的現狀，我們列出接下來開發者的修繕與優化落地重點並定義優先級：

### P0 優先級：啟動並驗證 YOLOv13 真實推理 (完成 Phase 4)
- **依賴條件**：您的實體計算環境需先通過 `uv sync --extra yolo` 成功搭載 `ultralytics` 包，並保證 `data/models/` 之中存放有效的訓練後權重檔（如 `yolov13n.pt`）。
- **具體行動**：
  1. 修改配置檔 `.env` 內 `INFERENCE_MODEL_MODE=yolov13`。
  2. 重啟推理服務。
  3. 自前端上傳實際包含雨霧情境的車輛相片，打通全流程。
  4. 確認 9000 端口沒有拋出 CUDA 或 CPU Tensor 異常，且前端完美貼合預測物件檢測框。

### P1 優先級：導入 Celery 異步處理 (推進 Phase 5)
- **依賴條件**：目前 Docker Compose 開起的 Redis 服務可以直充做為 Message Broker。
- **具體行動**：
  1. 在 Django 專案內部整合配置 `celery` worker。
  2. 重構原先在 `detection` view 內部使用同步阻塞等待 `POST /internal/inference/detect` 的片段為非同步任務 (`delay`)。
  3. 前端流程調整，變更為上傳請求後只返回 `task_id`，交由前端配合 `setInterval` 輪詢或是採用 WebSocket 監聽伺服器報告分析結束之狀態。

### P2 優先級：模型反饋與主動學習循環 (中長程規劃)
- **依賴條件**：實務營運產生模型效果瓶頸。
- **具體行動**：
  1. 在前端介面增加一個供管理員標記「檢測結果異常/需精修」的按鈕。
  2. 建立機制定期匯出這批圖片與對應 JSON。
  3. 將該批次資料回饋至 `docs/training-pipeline.md` 的離線標註環節，實現系統反覆迭代變強。
